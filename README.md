Во время тестов на локалке сохранение документа заниала ~ 45 сек, а генерация ответов 30-40 сек. Можно сокраить время в двое если испольщовать не локальный embeding модель.

Решение включает две главные вещи RAG И RLM.
RAG необходим для работы с данными в вектроном виде. В качестве векторной бд выступает qdrant (более понятный и быстрый по срвнению с другими).
RLM (Recursive Language Model) выступает сердцем. Используется для генерации ответа. Он получает на вход вопрос разбивает его на подзадачи и дальше по цепочке вызывет саму себя для решения подзадачи, но глубина рекурсии ограничениа либо ее вообще нет т.к мы контраоируем самые важные шаги.
То есть llm создала список необходиых задач для ответа на вопрос, а затем мы их в ручную собираем и по цепочке выплняем в ней же, моэно еще проверить ответ на каждом этапе есл ион не понравиться то выполнил подзадачу снова. Затем собрав все необходимые данные с подзадач, мы снова вызываем модльку уже для генерации финальлного ответа.
За счет всего этго можно добиться более высоко тчности ответов. 
Можно быо написать полноценый RLM без жесткого контроля, но тогда ответы и количество вызвов будет не предсказуемым (поскольку тогда RLM сама писала необхоимыый код).
Также есть графовое бд neo4j для храннеия связей между сущностями документов.
А для хранения метаданных использовался Postgresql

Сохранение документов происходит аснихронно за счет celery. Он по цепочке сохраняет в разлчиных бд. Сначала пасринг, потом разбиение на чанки (их храним в Postgres), извлчение сущностей (храним в бд), извлекаем связи (сохраняем в postgres), создаем векторное представлнеи чанков (сохраняем в qdrant) и потом создаем графы.

Для работы со всеи этим есть отдельные сервисы (всго их 7).
-document: crud операции с доками
-extraction: для извлечения сущностей
-graph: для работы с графами
-indexing: для индексации в вектрном бд
-parser: парсит доки различных форматов
-rlm: сама RLM (по сути обертка над LLM)
-search: для поиска во всех бд

В качестве временного фронта использовался streamlit
